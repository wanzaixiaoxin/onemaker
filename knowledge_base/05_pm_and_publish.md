# 05 项目管理与发行运营（PM & Publishing）

## 一、项目管理（Project Management）

### 职责定义
规划开发进度、分解任务、管理里程碑、控制风险。独立开发者更需要纪律性的自我管理。

### 关键产出物
- **项目计划（Project Plan）**：里程碑、时间线、资源分配
- **任务看板（Task Board）**：待办/进行中/完成
- **每周回顾**：进度总结与调整

### 详细操作步骤

#### Step 1: 制定里程碑计划
```
8阶段里程碑模板：

| 阶段 | 目标 | 产出物 | 时长参考 |
|------|------|-------|---------|
| 概念期 | 确定核心创意 | 概念文档、核心循环图 | 1-2周 |
| 原型期 | 验证核心玩法 | 可玩原型、核心机制验证 | 2-4周 |
| 垂直切片 | 一个完整关卡 | 高品质关卡Demo | 4-8周 |
| 生产期 | 全量内容制作 | 全部关卡/系统/资源 | 8-20周 |
| Alpha | 功能完整 | 功能完整版本 | 2-4周 |
| Beta | 内容完整+修复 | 内容完整版本 | 2-4周 |
| RC | 最终测试 | 候选发布版本 | 1-2周 |
| 发行 | 上架+运营 | 正式版+宣发 | 1周+持续 |

里程碑评审检查点：
每个阶段结束前必须回答：
1. 本阶段目标是否达成？
2. 是否有遗留的阻塞性问题？
3. 下一阶段的资源是否就绪？
4. 是否需要调整后续计划？
```

#### Step 2: 任务分解与看板
```
WBS（工作分解结构）模板：

游戏名称
├── 设计
│   ├── 概念文档
│   ├── GDD
│   ├── 关卡设计文档
│   └── 系统设计文档
├── 程序
│   ├── 核心框架
│   ├── 角色系统
│   ├── 战斗系统
│   ├── AI系统
│   ├── UI系统
│   └── 工具/管线
├── 美术
│   ├── 角色设计
│   ├── 场景设计
│   ├── UI设计
│   └── 特效设计
├── 音频
│   ├── BGM
│   ├── SFX
│   └── 语音
└── 发行
    ├── 商店素材
    ├── 宣传视频
    └── 社群运营

看板工具推荐：
| 工具 | 特点 | 适合 |
|------|------|------|
| Trello | 简单直观 | 小项目 |
| Notion | 全能工具 | 中项目 |
| GitHub Projects | 代码关联 | 程序员 |
| 飞书多维表格 | 免费 | 中文用户 |
```

#### Step 3: 风险管理
```
风险登记表模板：

| ID | 风险描述 | 概率 | 影响 | 等级 | 应对策略 |
|----|---------|------|------|------|---------|
| R01 | 核心玩法不好玩 | 中 | 高 | 红 | 尽早原型验证，2周内出Demo |
| R02 | 范围蔓延（越做越多） | 高 | 中 | 红 | 严格MVP，砍掉非核心 |
| R03 | 技术方案不可行 | 低 | 高 | 黄 | 提前做技术验证 |
| R04 | 动力丧失/倦怠 | 高 | 高 | 红 | 小步快跑，及时奖励自己 |
| R05 | 美术质量不达标 | 中 | 中 | 黄 | 用AI辅助，降低风格复杂度 |
| R06 | 上架审核被拒 | 低 | 中 | 黄 | 提前了解平台规则 |

风险等级判定：
  概率×影响 = 等级
  高×高 = 红色（必须应对）
  高×中 / 中×高 = 黄色（需要关注）
  其他 = 绿色（记录即可）
```

#### Step 4: 每周回顾流程
```
每周回顾模板（30分钟）：

1. 本周完成（5min）
   - [ ] 任务A
   - [ ] 任务B
   - 完成率：__%

2. 遇到的问题（10min）
   - 问题描述 → 根因分析 → 解决方案

3. 下周计划（10min）
   - [ ] 任务C（优先级P0）
   - [ ] 任务D（优先级P1）

4. 自我评估（5min）
   - 效率评分：__/10
   - 动力评分：__/10
   - 调整措施：____________

onemaker weekly 命令自动生成回顾模板
```

## 二、测试与质量保证（QA & Testing）

### 职责定义
系统性测试游戏的每个方面，确保功能正确、体验流畅、无致命Bug。

### 关键产出物
- **测试用例表**：功能测试、性能测试、兼容性测试
- **Bug追踪表**：Bug列表、优先级、状态
- **测试报告**：阶段性测试总结

### 详细操作步骤

#### Step 1: 测试用例编写
```
功能测试用例模板：

| 用例ID | 模块 | 测试内容 | 操作步骤 | 预期结果 | 实际结果 |
|--------|------|---------|---------|---------|---------|
| TC001 | 移动 | 角色行走 | 按住W键 | 角色向前移动 |  |
| TC002 | 移动 | 角色跳跃 | 按空格键 | 角色跳起并落地 |  |
| TC003 | 战斗 | 普通攻击 | 按鼠标左键 | 播放攻击动画+判定伤害 |  |
| TC004 | 战斗 | 受伤反馈 | 被敌人攻击 | HP减少+红色闪烁 |  |
| TC005 | UI | 暂停菜单 | 按ESC | 弹出暂停面板 |  |
| TC006 | 存档 | 保存进度 | 点击保存 | 数据写入文件 |  |
| TC007 | 存档 | 读取进度 | 点击读取 | 正确恢复游戏状态 |  |
| TC008 | 性能 | 帧率测试 | 复杂场景 | ≥30FPS |  |

必测类别：
- [ ] 核心玩法（移动/攻击/技能）
- [ ] 系统交互（背包/装备/商店）
- [ ] 关卡流程（开始→通关→下一关）
- [ ] 边界情况（空血/满级/无弹药）
- [ ] 存档/读档
- [ ] UI所有面板
- [ ] 设置面板所有选项
- [ ] 键鼠/手柄输入
```

#### Step 2: Bug分级与追踪
```
Bug严重度分级：

| 级别 | 名称 | 定义 | 示例 | 修复时限 |
|------|------|------|------|---------|
| S | 致命 | 崩溃/数据丢失/无法继续 | 闪退、存档损坏 | 立即 |
| A | 严重 | 核心功能异常 | 战斗无伤害、AI不动 | 当天 |
| B | 一般 | 功能异常但有替代方案 | 某个技能无特效 | 本周 |
| C | 轻微 | 视觉/UI小问题 | 文字错别字、对齐偏差 | 有空时 |
| D | 建议 | 体验优化建议 | 操作手感微调 | 版本迭代 |

Bug追踪表：
| ID | 标题 | 严重度 | 状态 | 发现日期 | 修复日期 | 备注 |
|----|------|-------|------|---------|---------|------|
| B001 | | S/A/B/C/D | 新/处理中/已修/已验 | | | |
```

#### Step 3: 性能测试
```
性能基准测试表：

| 测试场景 | 目标FPS | 实际FPS | Draw Call | 三角面 | 内存 | 通过 |
|---------|---------|---------|----------|-------|------|------|
| 主菜单 | 60 | | | | | |
| 村庄场景 | 60 | | | | | |
| 战斗(5敌) | 30 | | | | | |
| Boss战 | 30 | | | | | |

性能预算（PC）：

- FPS：≥30

- 包体大小：≤500MB

## 三、发行准备（Publishing Preparation）

### 职责定义
准备上架所需的所有素材、文档、合规文件。

### 关键产出物
- **商店页面素材**：截图、视频、描述文案
- **合规文件**：隐私政策、年龄分级
- **宣传素材**：预告片、宣传图

### 详细操作步骤

#### Step 1: 商店素材清单
```
Steam商店素材清单：

| 素材 | 规格 | 数量 | 优先级 |
|------|------|------|-------|
| 商店图标 | 32x32 PNG | 1 | P0 |
| 头像/Logo | 460x215 PNG | 1 | P0 |
| 商店封面 | 600x900 PNG | 1 | P0 |
| 页面头图 | 1920x622 PNG | 1 | P0 |
| 截图 | 1920x1080 PNG | 5-15 | P0 |
| 宣传视频 | MP4, ≤5min | 1-3 | P0 |
| 胶囊图(Capsule) | 231x87/616x353 | 各1 | P0 |
| 背景图 | 1920x1080+ | 1 | P1 |

商店描述文案模板：
1. 简短描述（一句话卖点，≤150字）
2. 详细描述：
   - 游戏概述（2-3句）
   - 核心特色（3-5条，每条配图）
   - 游戏系统说明
   - 配置要求（最低/推荐）
3. 标签选择：5-15个
```

#### Step 2: 平台上架流程
```
主要PC平台对比：

| 平台 | 抽成 | 审核周期 | 适合 | 流程 |
|------|------|---------|------|------|
| Steam | 30%→20% | 1-5天 | PC首选 | Steamworks注册→缴费→页面→审核 |
| Epic Games Store | 12% | 1-7天 | PC | 开发者门户申请 |
| GOG | 30% | 较长 | 品质游戏 | 邮件申请 |
| itch.io | 0-10% | 即时 | 独立/实验 | 直接上传 |

Steam上架步骤：
1. 注册Steamworks开发者账号（$100费用）
2. 创建App，填写商店信息
3. 上传构建包（Depot）
4. 设置定价和地区
5. 提交审核（通常1-5个工作日）
6. 审核通过→设置发布日期
```

#### Step 3: 定价策略
```
定价参考表（独立游戏）：

| 游戏规模 | 开发时长 | 建议价格(CNY) | 建议价格(USD) |
|---------|---------|-------------|-------------|
| 小品级（<2h） | 1-3月 | 18-38 | $2.99-$4.99 |
| 中等（2-8h） | 3-6月 | 38-68 | $4.99-$9.99 |
| 大作（8h+） | 6-12月 | 68-128 | $9.99-$19.99 |
| 精品独立 | 12月+ | 128-198 | $19.99-$29.99 |

定价注意事项：
- 参考同类游戏定价
- 首发折扣（-10%~-20%）
- 地区定价差异化
- 季度/节日促销规划
```

## 四、营销推广（Marketing）

### 职责定义
制定营销计划、建设社群、制作宣传内容、联系媒体/KOL。

### 关键产出物
- **营销日历**：从开发到发行后的完整推广时间线
- **社群资产**：社交媒体账号、Discord/QQ群
- **宣传素材**：预告片、GIF、开发日志

### 详细操作步骤

#### Step 1: 营销时间线
```
独立游戏营销时间线：

发行前6个月：
- [ ] 建立社交媒体账号（B站/微博/TapTap/Twitter）
- [ ] 发布首支预告片/概念图
- [ ] 建立社群（QQ群/Discord）

发行前3个月：
- [ ] 每周发布开发日志
- [ ] 在TapTap/Steam创建商店页面
- [ ] 开始收集愿望单/预约
- [ ] 联系游戏媒体/KOL

发行前1个月：
- [ ] 发布正式预告片
- [ ] 发放测试Key给KOL
- [ ] 准备首发日素材
- [ ] 制定发行日直播计划

发行日：
- [ ] 全平台发布公告
- [ ] B站/抖音直播
- [ ] 社群活动（抽奖/限时福利）

发行后持续：
- [ ] 每周更新/开发日志
- [ ] 收集并回应玩家反馈
- [ ] 规划DLC/更新内容
```

#### Step 2: 社交媒体运营
```
平台运营策略：

| 平台 | 内容形式 | 频率 | 目标 |
|------|---------|------|------|
| B站 | 开发日志/实机演示 | 每周1次 | 曝光+核心粉丝 |
| 抖音/快手 | 短视频/精彩片段 | 每周2-3次 | 广泛传播 |
| TapTap | 开发日志/社区互动 | 每周1次 | 预约+口碑 |
| 微博 | 图文/抽奖 | 每周2-3次 | 中文社区 |
| Twitter/X | GIF/截图 | 每日 | 海外曝光 |
| 小红书 | 美术/设计分享 | 每周1-2次 | 女性用户 |

开发日志内容模板：

2. 遇到的挑战和解决方案
3. 下周计划
4. 互动话题（征集意见/投票）
```

#### Step 3: KOL/媒体合作
```
KOL合作清单：

联系前准备：
- [ ] 游戏Demo/测试版
- [ ] Press Kit（媒体包）
- [ ] 精选截图和GIF
- [ ] 一句话描述+核心卖点

Press Kit内容：
1. 游戏简介（100字/300字版本）
2. 核心特色（3-5条）
3. 开发者简介
4. 高清截图（10-20张）
5. 预告片链接
6. Logo（多尺寸）
7. 联系方式

KOL筛选标准：
- 粉丝量：1K-100K（性价比最高）
- 内容方向：匹配游戏类型
- 互动率：>3%
- 历史视频质量

## 五、运营与更新（LiveOps）

### 职责定义
发行后的持续运营：更新维护、社群管理、数据分析、DLC规划。

### 关键产出物
- **更新计划**：版本迭代规划
- **数据看板**：销量、在线、评价追踪
- **社群管理**：玩家反馈收集与回应

### 详细操作步骤

#### Step 1: 发行后第一周
```
发行首周紧急清单：

Day 0（发行日）：
- [ ] 确认所有平台正常上架
- [ ] 监控崩溃报告
- [ ] 准备Day1 Patch

Day 1-3：
- [ ] 收集玩家反馈
- [ ] 修复S/A级Bug
- [ ] 发布首日热修复
- [ ] 回复商店评论

Day 4-7：
- [ ] 发布首周更新说明
- [ ] 优化高频反馈问题
- [ ] 发布社区公告
- [ ] 分析首周数据
```

#### Step 2: 数据追踪
```
关键数据指标：

| 指标 | 计算方式 | 健康值 |
|------|---------|-------|
| 愿望单转化率 | 购买/愿望单 | >10% |
| 首周销量 | 发行7天内 | 看类型 |
| 好评率 | 好评/总评 | >80% |
| 平均游玩时长 | Steam统计 | >游戏时长的50% |
| 退款率 | 退款/购买 | <10% |
| 日活(DAU) | 每日活跃玩家 | 趋势上升 |

数据记录表：
| 日期 | 销量 | 累计 | 好评率 | DAU | 在线峰值 | 备注 |
|------|------|------|-------|-----|---------|------|
```

#### Step 3: 更新节奏规划
```
版本更新节奏：

热修复（Hotfix）：随时
- S级Bug修复
- 崩溃修复
- 不改版本号

小更新（Patch）：每1-2周
- Bug修复
- 小功能优化
- 版本号+0.0.1

功能更新（Update）：每1-2月
- 新内容/新功能
- 平衡性调整
- 版本号+0.1

大版本（Expansion）：每3-6月
- 大量新内容
- 新系统/新模式
- 版本号+1.0

更新公告模板：
1. 新增内容
2. 改进优化
3. Bug修复
4. 已知问题
5. 下次更新预告
```
