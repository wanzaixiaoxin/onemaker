# 05 项目管理与发行运营（PM & Publishing）

## 一、项目管理（Project Management）

### 职责定义
规划开发进度、分解任务、管理里程碑、控制风险。独立开发者更需要纪律性的自我管理。

### 关键产出物
- **项目计划（Project Plan）**：里程碑、时间线、资源分配
- **任务看板（Task Board）**：待办/进行中/完成
- **周报/日志**：进度记录和复盘

### 独立游戏开发阶段划分

```
阶段1: 原型期（Prototype）    [2-4周]
  └─ 验证核心玩法是否有趣

阶段2: 垂直切片（Vertical Slice） [4-8周]
  └─ 完整体验一个关卡/循环的全部内容

阶段3: 量产期（Production）   [3-12个月]
  └─ 批量制作内容（关卡、资产、系统）

阶段4: Alpha                 [2-4周]
  └─ 功能完整，开始内部测试

阶段5: Beta                  [2-4周]
  └─ 内容完整，修复Bug，优化性能

阶段6: 发布候选（RC）         [1-2周]
  └─ 最终测试，准备上架

阶段7: 发布（Launch）         [1周]
  └─ 正式上架，开始运营
```

### 任务管理工具
| 工具 | 特点 | 推荐度 |
|------|------|-------|
| Notion | 全能文档+看板+数据库 | ★★★★★ |
| Trello | 简单看板，免费 | ★★★★ |
| Linear | 开发者友好的项目管理 | ★★★★ |
| GitHub Projects | 与代码仓库集成 | ★★★★ |
| 飞书/钉钉 | 国内生态 | ★★★ |

### 独立开发者项目管理原则
1. **最小可玩优先**：先做能玩的，再做好看的
2. **时间盒（Timebox）**：每个任务限定时间，超时就简化
3. **每周复盘**：回顾完成了什么、卡在哪里、下周重点
4. **砍功能勇气**：Scope Creep是独立开发的最大杀手
5. **公开进度**：在社交媒体分享开发进度，建立社区

### 里程碑模板
```
M1 - 核心原型可玩          [ ]
M2 - 美术风格确定          [ ]
M3 - 垂直切片完成          [ ]
M4 - 第一个完整关卡        [ ]
M5 - 所有核心系统完成      [ ]
M6 - Alpha（功能完整）     [ ]
M7 - Beta（内容完整）      [ ]
M8 - RC（发布候选）        [ ]
M9 - 正式发布              [ ]
```

### AI协作方式
- **项目规划**：描述游戏规模，让AI估算阶段和里程碑
- **任务分解**：描述一个大任务，让AI拆分为可执行的小任务
- **风险评估**：让AI分析项目计划中的潜在风险
- **周报生成**：提供本周完成内容，让AI生成周报

## 二、QA测试（Quality Assurance）

### 职责定义
确保游戏功能正确、性能达标、无崩溃、多平台兼容。

### 关键产出物
- **测试用例**：覆盖所有功能的测试场景
- **Bug清单**：优先级分类的Bug列表
- **性能报告**：FPS、内存、加载时间的基准数据
- **兼容性报告**：不同硬件/系统的运行情况

### 测试类型
| 类型 | 关注点 | 方法 |
|------|-------|------|
| 功能测试 | 功能是否按设计工作 | 手动测试 + 自动化 |
| 性能测试 | FPS/内存/加载时间 | Profiler + 目标设备测试 |
| 兼容性测试 | 不同硬件/系统是否正常 | 多设备覆盖 |
| 用户体验测试 | 玩家是否理解/享受 | 外部测试者 |
| 回归测试 | 修复Bug是否引入新问题 | 已有测试用例重跑 |

### Bug优先级定义
| 等级 | 定义 | 修复时限 |
|------|------|---------|
| P0 - 阻断 | 崩溃、存档丢失、无法继续 | 立即 |
| P1 - 严重 | 核心功能异常、严重卡顿 | 当天 |
| P2 - 一般 | 非核心功能异常、表现问题 | 本周 |
| P3 - 轻微 | 错别字、UI对齐、小优化 | 有空时 |

### 自动化测试策略
```
单元测试 → 测试单个函数/类
集成测试 → 测试模块间交互
端到端测试 → 测试完整游戏流程
性能基准测试 → 自动运行性能测试
```

### AI协作方式
- **测试用例生成**：描述功能，让AI生成测试用例
- **Bug分类**：提供Bug描述，让AI分析可能原因
- **自动化脚本**：让AI编写自动化测试脚本
- **性能分析**：提供Profiler数据，让AI分析瓶颈

## 三、发行上架（Publishing）

### 职责定义
准备上架材料、对接平台、通过审核、发布游戏。

### 主要发布平台
| 平台 | 适合类型 | 分成 | 上架难度 |
|------|---------|------|---------|
| Steam | PC游戏 | 70/30 | 中（Steam Direct $100） |
| itch.io | 独立/实验性 | 可自定义 | 低 |
| Epic Games Store | PC游戏 | 88/12 | 中 |
| TapTap | 国内移动端 | 50/50~70/30 | 中 |
| App Store | iOS | 70/30 | 高（年费$99） |
| Google Play | Android | 70/30 | 中（$25一次性） |
| 微信小游戏 | 国内小游戏 | 按流水阶梯 | 中 |

### 上架材料清单
- [ ] 游戏构建包（多平台）
- [ ] 商店页面素材
  - [ ] 游戏图标（多尺寸）
  - [ ] 商店头图/Banner
  - [ ] 截图（至少5张，含不同场景）
  - [ ] 宣传视频/Trailer（1-2分钟）
  - [ ] 游戏描述（短描述+长描述）
- [ ] 年龄评级/内容审查
- [ ] 隐私政策
- [ ] 用户协议
- [ ] 测试报告

### Steam上架流程
```
1. 注册Steamworks开发者（$100）
2. 创建商店页面
3. 上传素材（图标、截图、视频）
4. 填写商店信息（描述、标签、系统要求）
5. 上传构建包到SteamPipe
6. 提交审核（通常1-5个工作日）
7. 审核通过 → 设置发布日期
8. 发布！
```

### AI协作方式
- **商店文案撰写**：让AI写游戏描述、特色卖点
- **标签建议**：让AI推荐Steam标签
- **Trailer脚本**：让AI设计宣传视频的分镜脚本
- **FAQ编写**：让AI生成常见问题解答
- **法律文档模板**：让AI生成隐私政策/用户协议草案（需律师审核）

## 四、营销运营（Marketing & Operations）

### 职责定义
让目标玩家知道你的游戏、建立社区、收集反馈、持续运营。

### 营销时间线
```
开发早期（原型期）
  └─ 开始在社交媒体分享开发日志
  └─ 建立Wishlist页面（Steam）

开发中期（量产期）
  └─ 定期发布开发进度（截图/短视频）
  └─ 参加游戏展（线上/线下）
  └─ 联系游戏媒体/KOL

发布前2-3个月
  └─ 发布Trailer
  └─ 开放Demo（Steam Next Fest）
  └─ 媒体评测包
  └─ KOL评测邀请

发布前2周
  └─ 发布倒计时
  └─ 社区活动
  └─ 最终Trailer

发布日
  └─ 发布公告
  └─ 社区互动
  └─ 直播

发布后
  └─ 收集反馈
  └─ 修复Bug
  └─ 内容更新计划
```

### 社区运营渠道
| 渠道 | 适合内容 | 投入产出比 |
|------|---------|-----------|
| Twitter/X | 开发截图、GIF | ★★★★★ |
| B站 | 开发日志、实机演示 | ★★★★★（国内） |
| 小红书 | 美术分享、幕后花絮 | ★★★★（国内） |
| Discord | 社区建设、反馈收集 | ★★★★ |
| Reddit | 开发讨论、社区互动 | ★★★ |
| YouTube | 开发日志、Trailer | ★★★ |
| 微博/公众号 | 国内宣传 | ★★★ |

### 关键营销指标
- **Wishlist数量**（Steam最重要的指标）
- **社交媒体关注/互动**
- **Demo下载量和转化率**
- **媒体/KOL覆盖率**
- **首日/首周销量**

### AI协作方式
- **营销文案**：让AI写社交媒体帖子、新闻稿
- **内容规划**：让AI制定社交媒体发布日历
- **邮件模板**：让AI写KOL/媒体联系邮件
- **数据分析**：让AI分析营销数据并给出建议
- **社区管理**：让AI起草社区公告和回复模板
